# 3D Data and Tiling

muDM supports 3D mesh data through dedicated geometry types, a Rust-accelerated tiling pipeline, and a Three.js-based web viewer. This page covers the full 3D workflow: defining 3D geometries, ingesting mesh data, generating tiled output in multiple formats, and visualizing the results.

## 3D Geometry Types

muDM extends GeoJSON with two 3D mesh geometry types based on ISO 19107.

### TIN (Triangulated Irregular Network)

A triangle mesh surface. Each face is a closed ring of exactly 4 positions (3 vertices + repeated first vertex). This is the primary geometry type used for tiled 3D mesh output.

```json
{
    "type": "Feature",
    "geometry": {
        "type": "TIN",
        "coordinates": [
            [[[0, 0, 0], [10, 0, 0], [5, 8, 3], [0, 0, 0]]],
            [[[10, 0, 0], [10, 8, 0], [5, 8, 3], [10, 0, 0]]],
            [[[0, 0, 0], [0, 8, 0], [5, 8, 3], [0, 0, 0]]]
        ]
    },
    "properties": {
        "featureClass": "neuron",
        "cellType": "pyramidal",
        "region": "CA1"
    }
}
```

### PolyhedralSurface

A closed surface mesh consisting of polygonal faces (not limited to triangles). Each face follows the same structure as a Polygon.

```json
{
    "type": "Feature",
    "geometry": {
        "type": "PolyhedralSurface",
        "coordinates": [
            [[[0, 0, 0], [10, 0, 0], [10, 10, 0], [0, 10, 0], [0, 0, 0]]],
            [[[0, 0, 5], [10, 0, 5], [10, 10, 5], [0, 10, 5], [0, 0, 5]]],
            [[[0, 0, 0], [10, 0, 0], [10, 0, 5], [0, 0, 5], [0, 0, 0]]],
            [[[0, 10, 0], [10, 10, 0], [10, 10, 5], [0, 10, 5], [0, 10, 0]]],
            [[[0, 0, 0], [0, 10, 0], [0, 10, 5], [0, 0, 5], [0, 0, 0]]],
            [[[10, 0, 0], [10, 10, 0], [10, 10, 5], [10, 0, 5], [10, 0, 0]]]
        ]
    },
    "properties": {
        "featureClass": "organelle"
    }
}
```

Both types support a `tiles` field as an alternative to inline `coordinates`, referencing external tiled data (e.g., a 3D Tiles tileset).

## StreamingTileGenerator API

The `StreamingTileGenerator` class (from `microjson._rs`) is the main entry point for 3D tiling. It uses octree-based spatial indexing with parallel processing via rayon.

### Constructor

```python
from microjson._rs import StreamingTileGenerator

gen = StreamingTileGenerator(
    min_zoom=0,         # Minimum zoom level
    max_zoom=4,         # Maximum zoom level
    extent=4096,        # XY tile extent in integer coordinates
    extent_z=4096,      # Z tile extent in integer coordinates
    buffer=0.0,         # Tile buffer (fraction of tile size)
    base_cells=10,      # Grid resolution at zoom 0 for vertex clustering
    temp_dir="/data/tmp",  # Temp directory for fragment files
    num_buckets=256,    # Buckets for redistribution (see below)
)
```

**Key parameters:**

| Parameter | Default | Description |
|-----------|---------|-------------|
| `base_cells` | 10 | Grid resolution for QEM simplification at zoom 0. Use 10 for solid regions, 50-200 for thin branching structures (neurons). |
| `temp_dir` | system temp | Directory for intermediate fragment files. Use a large volume for big datasets. |
| `num_buckets` | 256 | Number of buckets for redistribution. Increase (512-1024) for datasets that exceed RAM. |

### Ingesting OBJ Meshes

```python
# Single file
feature_id = gen.add_obj_file(
    "neuron_001.obj",
    bounds=(xmin, ymin, zmin, xmax, ymax, zmax),
    tags={"cellType": "pyramidal", "region": "CA1"},
)

# Batch parallel ingestion (uses rayon thread pool)
feature_ids = gen.add_obj_files(
    paths=["neuron_001.obj", "neuron_002.obj", ...],
    bounds=(xmin, ymin, zmin, xmax, ymax, zmax),
    tags_list=[{"cellType": "pyramidal"}, {"cellType": "basket"}, ...],
    ingest_threads=0,  # 0 = auto (all cores)
)
```

Batch ingestion uses per-thread fragment shards with no write locks, scaling linearly with core count.

### Ingesting from Parquet

For pre-processed mesh data, bypass OBJ parsing entirely:

```python
count = gen.add_parquet_meshes(
    "meshes.parquet",
    positions_col="positions",   # large_binary: f32 LE [x,y,z,x,y,z,...]
    indices_col="indices",       # large_binary: u32 LE [i0,i1,i2,...]
    prop_col="cell_type",        # optional property column
    prop_name="cellType",        # output tag name
    layer_type="neurons",        # layer_type tag value
    bounds=(0, 0, 0, 1, 1, 1),
)
```

## Output Formats

After ingesting data, generate output in one or more formats:

### 3D Tiles (GLB)

OGC 3D Tiles with optional mesh compression. Produces a `tileset.json` manifest and per-tile `.glb` files.

```python
n_tiles = gen.generate_3dtiles(
    "output/3dtiles",
    bounds,
    compression="meshopt",      # "meshopt", "draco", or "none"
    max_concurrent_buckets=8,   # parallel bucket processing
    max_memory_gb=0,            # 0 = auto-detect from /proc/meminfo
)
```

### Tiled Parquet

ZSTD-compressed columnar format partitioned by zoom level. Ideal for ML training pipelines.

```python
n_rows = gen.generate_parquet_native(
    "output/features.parquet",
    bounds,
    simplify=True,
    compression="zstd",
)
```

Output structure: `features.parquet/zoom={z}/part_*.parquet`

### PBF3

Custom protobuf format for compact 3D tile transfer:

```python
n_tiles = gen.generate_pbf3("output/pbf3", layer_name="neurons")
```

### Compression Comparison

| Compression | Encode Speed | Browser Decode | Size (per brain) | With Brotli | Lossless |
|-------------|-------------|----------------|-------------------|-------------|----------|
| `"meshopt"` (default) | ~2 min | ~1 GB/s | 82 MB | 57 MB | Yes |
| `"draco"` | ~6.5 min | ~50-100 MB/s | 47 MB | N/A | No (quantizes) |
| `"none"` | Fastest | N/A | 232 MB | N/A | Yes |

**Recommendation:** Use meshopt for web delivery. With Brotli HTTP transport compression (enabled by the viewer server), meshopt GLBs approach Draco sizes while decoding ~100x faster in the browser.

## Bucketed Redistribution

For datasets that exceed available RAM (thousands of neurons, hundreds of GB of fragments), the pipeline uses a two-phase bucketed redistribution strategy:

**Phase 1: Shard → Bucket redistribution.** Each ingestion shard is read on a separate thread. Fragments are hashed by tile key (`hash(z, x, y, d) % num_buckets`) and written to per-shard bucket files. No locks required.

**Phase 2: Bucket processing.** Each bucket is loaded into memory and processed independently. Tile encoding within each bucket runs in parallel via rayon. Memory per bucket is strictly bounded.

**Tuning `num_buckets`:**

- Smaller values = larger per-bucket memory, fewer I/O operations
- Larger values = smaller per-bucket memory, more fragmented I/O
- Default (256) works for most datasets with 8+ GB RAM
- For very large datasets (5000+ neurons), use 512-1024

The memory budget is auto-detected from `/proc/meminfo` (80% of total RAM) or can be set via `max_memory_gb`.

## 3D Viewer

The Three.js-based viewer provides interactive visualization of 3D Tiles output.

### Starting the Viewer

```bash
python viewer/serve.py --tiles-base data/tiles --port 8080
# Open http://localhost:8080/
```

The server:
- Serves GLB tiles with on-the-fly Brotli or gzip Content-Encoding (512 MB cache)
- Serves Neuroglancer precomputed format via `/neuroglancer/` prefix
- Reads `pyramids.json` manifest from the tiles-base directory

### Viewer Features

- **3D Tiles LOD**: Automatic level-of-detail management based on camera distance
- **Feature selection**: Search and filter by feature ID, toggle individual features via checkboxes
- **Color-by attributes**: Categorical and numeric coloring from feature properties
- **Slice plane**: Interactive cross-section for examining internal structures
- **Orbit camera**: Z-up coordinate system matching muDM conventions
- **Axis gizmo**: Visual coordinate reference
- **Scale bar**: Micron-unit scale display
- **Overview panel**: Spatial filtering with ring selection
- **Info panel**: Hover/click to inspect feature properties
